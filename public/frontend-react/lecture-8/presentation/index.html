Part 1 — Hoisting

Hoisting: What Actually Happens

JavaScript moves var declarations to the top of their scope before execution.

What you write:

console.log(score); // ?
var score = 100;
console.log(score); // ?

What JS actually runs:

var score; // hoisted to top
console.log(score); // undefined
score = 100;
console.log(score); // 100
🔍 Parse JS scans all var declarations
→
⬆ Hoist Moves to top as undefined
→
▶ Execute Assigns value when line runs
⚠ let and const are also hoisted, but land in the Temporal Dead Zone (TDZ) — accessing them before declaration throws a ReferenceError, not undefined.

Part 2 — Type Inference

JavaScript Type Inference (Implicit Typing)

You don't declare types — you assign values. JS figures out the rest.

let x = 5;          // JS infers: number
let city = "Lahore";   // JS infers: string
let active = true;     // JS infers: boolean
keyword variable name assignment value (determines type) 
// Type can change at runtime — no warning!
let x = 5;
console.log(typeof x);  // "number"

x = "hello";
console.log(typeof x);  // "string" — same variable, different type!

x = true;
console.log(typeof x);  // "boolean" — JS allows this silently
⚠ Powerful but risky: JS never warns you when a variable changes type. This is exactly the problem TypeScript solves — it freezes the inferred type and alerts you the moment you break it.

Part 2 — Type Conversion

Converting Types

prompt() always returns a string — you must convert for math.

String → Number

const input = prompt("Enter price:");
// input is always a string!

const price = Number(input);   // explicit
const price2 = +input;         // shorthand
const price3 = parseInt(input); // integer only
const price4 = parseFloat(input); // with decimals

console.log(typeof price); // "number"

Number → String

const total = 199.99;

const s1 = String(total);   // "199.99"
const s2 = total.toString(); // "199.99"
const s3 = `${total}`;      // "199.99" (template)

// Coercion trap!
console.log("5" + 3); // "53" not 8!
⚠ Type Coercion Trap: "5" + 3 = "53" (string concatenation). "5" - 3 = 2 (numeric subtraction). The + operator checks types first. This is why TypeScript exists.

Part 3 — Equality

Loose == vs Strict ===

😄 Loose Equality ==

Converts types before comparing. Surprises await.

5    ==  "5"    // true ⚠️
0    ==  false  // true ⚠️
null == undefined // true ⚠️
[]   ==  false  // true ⚠️
""   ==  false  // true ⚠️
VS

✅ Strict Equality ===

No type conversion. Same value AND same type.

5    === "5"    // false ✅
0    === false  // false ✅
null === undefined // false ✅
[]   === false  // false ✅
""   === false  // false ✅

Truthy & Falsy

Every value in JavaScript has an implicit boolean nature. Conditions don't require actual booleans.

✅ Truthy

Values that behave like true in a condition.

Everything that is NOT falsy.

if (42)      { /* runs */ }
if ([])      { /* runs */ }
if ({})      { /* runs */ }
VS

❌ Falsy

Only 6 values behave like false. Everything else is truthy.

if (false)     { /* skipped */ }
if (0)         { /* skipped */ }
if ("")        { /* skipped */ }
if (null)      { /* skipped */ }
if (undefined) { /* skipped */ }
if (NaN)       { /* skipped */ }

Part 4 — Truthy Surprises

Truthy Values That Surprise Beginners

// These are all TRUTHY — they run the if block
if ("0")     { console.log("'0' is truthy!"); }  // non-empty string
if ("false") { console.log("'false' is truthy!"); } // non-empty string
if ([])      { console.log("[] is truthy!"); }   // empty array
if ({})      { console.log("{} is truthy!"); }   // empty object
if (-1)      { console.log("-1 is truthy!"); }   // non-zero number
if (Infinity){ console.log("Infinity is truthy!"); }
⚠ The Most Common Bug: if (myArray) is always true even when the array is empty! To check if an array has items, use if (myArray.length > 0) or if (myArray.length).

Part 4 — Practical Patterns

Truthy/Falsy in Real Code

// ✅ Default value pattern (|| operator)
const username = prompt("Your name:") || "Guest";
// If prompt returns "" (empty), use "Guest"

// ✅ Guard check pattern (&& operator)
const user = null;
const name = user && user.name; // null (doesn't crash!)
// If user is falsy, stop — don't access .name

// ✅ Boolean conversion
const hasDiscount = Boolean(discountCode); // true if truthy
const hasDiscount2 = !!discountCode;       // same, shorter

// ✅ Existence check
const price = 0;
if (price !== undefined) {  // CORRECT: price=0 is valid!
    console.log("Price set:", price);
}
// if (price) { ... } would FAIL — 0 is falsy!

Part 4 — Summary

Truthy & Falsy — Key Takeaways

  • Common Mistake: Checking if (array) to see if an array has items — use if (array.length)
  • Common Mistake: Using if (price) when price can legitimately be 0 — use if (price !== undefined)
  • Best Practice: Use || for default values, && for guard checks — these patterns appear in every React component
  • Best Practice: When unsure, use Boolean(value) or !!value to explicitly see what a value coerces to

Part 5 — Type Annotations

Declaring Types in TypeScript

// Primitive type annotations
let age: number = 25;
const name: string = "Rana";
let isActive: boolean = true;

// TypeScript infers types automatically
let score = 100;        // inferred: number
let city = "Lahore";    // inferred: string

// Special types
let anything: any = 42;        // disables type checking (avoid!)
let safer: unknown = "hello";  // must check type before use

// Arrays
let prices: number[] = [10, 20, 30];
let tags: string[] = ["react", "typescript"];
💎 Pro insight: TypeScript's inference is so good that you rarely need to write type annotations for variables. You do need them for function parameters — that's where the real value is.

Part 5 — Type Inference

TypeScript Type Inference

TS infers types from the initial assignment — annotation is optional for variables.

let — Widened Type

let status = "active";
// inferred: string

status = "inactive"; // ✅ OK
status = 42;
// ❌ Error: Type 'number'
// not assignable to 'string'
VS

const — Literal Type

const status = "active";
// inferred: "active" (literal!)

// TS knows it can ONLY be "active"
// Never changes — most specific type
⚡ When MUST you annotate? Function parameters, empty arrays (let tags: string[] = []), and ambiguous return types. For simple variable assignments, let TS infer.
💎 Production rule: Let TS infer for variables; always annotate function parameters. TypeScript's inference is smarter than most developers realize.

✅ TypeScript — Caught Immediately

const price: number = "199";
// ❌ ERROR at line 1:
// Type 'string' is not assignable
// to type 'number'.
// Fix it before you can run code!

Part 5 — Summary

TypeScript Preview — Key Takeaways

  • Common Mistake: Using any everywhere — it completely disables TypeScript's protection. Use unknown instead, then narrow with type checks.
  • Common Mistake: Writing explicit type annotations for every variable — trust TypeScript's inference for simple assignments

Lecture 8 — Cheat Sheet

Variables

  • Default to const; use let to reassign
  • Never use var in modern code
  • var is hoisted as undefined
  • let/const in TDZ before declaration

Data Types

  • 7 primitives: string number boolean null undefined symbol bigint
  • typeof null === "object" — historic bug
  • Always Number() convert prompt() input
  • Primitives by value, objects by reference

Operators

  • Always use === and !==
  • Precedence: () > ** > */% > +-
  • Ternary: condition ? true : false
