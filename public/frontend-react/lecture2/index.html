Part 1

Why Semantic HTML Matters

HTML elements carry meaning — not just appearance

  • In Lecture 1, we used <h1>, <p>, <ul> — these have meaning
  • But what about the sections of a page? Header? Navigation? Footer?
  • Many websites wrap everything in <div> tags — "div soup"

The Moving House Analogy

📦 Unmarked Bags

📦 Labeled Boxes

Kitchen, Bedroom, Office

Div Soup vs Semantic HTML

❌ Non-Semantic


<div id="header">
  <div id="nav">...</div>
</div>
<div id="content">...</div>
<div id="footer">...</div>
VS

✅ Semantic


<header>
  <nav>...</nav>
</header>
<main>...</main>
<footer>...</footer>
💡 Both render identically — but semantic HTML has meaning for screen readers, search engines, and developers.

Concept Check

Both render identically in the browser. So why should we use <header> instead of <div id="header">?

  1. <header> loads faster in the browser
  2. Screen readers, search engines, and developers can understand the page structure
  3. <div> is deprecated in HTML5
  4. <header> automatically adds CSS styling

Part 1 Summary

  • Common Mistake: Using <div> for everything because "it works" — visually yes, semantically no
  • Best Practice: Replace <div> with semantic elements whenever the content has a role
  • Pro Tip: "The biggest red flag in code reviews? Div soup. Don't be that developer."

Spot the Error


<header>
    <h1>My Portfolio</h1>
</header>
<main>
    <h2>About</h2>
</main>
<main>
    <h2>Projects</h2>
</main>

What's wrong with this code?

  1. <h1> should be outside <header>
  2. Two <main> elements — only ONE allowed per page
  3. <header> is missing a <nav>
  4. Nothing is wrong

Part 2 Summary

  • Common Mistake: Using <section> as a generic wrapper — that's <div>'s job
  • Best Practice: One <main> per page. Ask: "Does this content have a role?" If yes → semantic element. If no → <div>.
  • Pro Tip: "When I interview developers, I always look at their HTML structure. Semantic HTML shows understanding."

Accessibility (A11y) Fundamentals

Making websites usable by everyone

The Curb Cut Effect

Ramps weren't just built for wheelchairs — they help everyone

♿ Wheelchair users
👶 Parents with strollers
💼 Delivery workers
🦽 Travelers
💎 Accessible websites help keyboard users, slow connections, mobile users, search engines — and yes, screen readers too.

Accessibility Workflow

→

🚀 Ship It Accessible for all
⚠ Legal fact: Inaccessible websites can lead to lawsuits (ADA in the US, EAA in the EU). This is real.

Semantic Elements Create Landmarks

HTML Element ARIA Landmark Screen Reader Announces
<header> banner "Banner landmark"
<nav> navigation "Navigation landmark"
<main> main "Main landmark"
<aside> complementary "Complementary landmark"
<footer> contentinfo "Content information"
💡 Use semantic HTML first. You get landmarks for free — no extra ARIA needed!

ARIA: When Semantic HTML Isn't Enough


<!-- Add clarity to navigation -->
<nav aria-label="Main navigation">...</nav>

<!-- Label with an existing heading -->
<section aria-labelledby="about-heading">
    <h2 id="about-heading">About Me</h2>
</section>

<!-- Accessible link (explains action) -->
<a href="https://linkedin.com" target="_blank"
   aria-label="Visit LinkedIn profile (opens in new tab)">
   LinkedIn
</a>
⚠ First rule of ARIA: Don't use ARIA if a native HTML element can do the job. Semantic first!

Spot the Problem


<img src="team-photo.jpg" alt="">

The developer added alt="". Is this accessible?

  1. Yes — alt is present, that's all that matters
  2. No — empty alt="" marks the image as decorative (skipped by screen readers)
  3. No — alt should always say "image"
  4. It depends — empty alt is correct only for decorative images

Part 3 Summary

  • Common Mistake: Empty alt="" on informational images — screen readers skip them entirely
  • Best Practice: Test with keyboard only (Tab key). If you can't reach everything, it's not accessible.
  • Pro Tip: "Screen readers are not edge cases. Google's crawler is essentially a screen reader. Good A11y = good SEO."

💎 "Semantic HTML is the easiest SEO win — and it's free. No tools, no plugins."

Essential SEO Meta Tags


<head>
  <!-- Browser tab + search results title -->
  <title>Ayesha Khan — Frontend Developer</title>

  <!-- Search result snippet (~155 chars) -->
  <meta name="description"
    content="Portfolio of Ayesha Khan, specializing
    in React and Next.js.">

  <!-- Social sharing (WhatsApp, Facebook) -->
  <meta property="og:title" content="Ayesha Khan">
  <meta property="og:description" content="...">
  <meta property="og:image" content="preview.jpg">
</head>

Predict the Output

You share this page on WhatsApp. What title and description will appear in the preview?

  1. The <h1> and first <p> from the page
  2. The og:title and og:description from the meta tags
  3. The URL of the page
  4. Nothing — WhatsApp can't read HTML

Part 4 Summary

  • Common Mistake: Missing meta description — Google auto-generates one (usually poorly)
  • Best Practice: Every page should have unique <title> and <meta description>. Keep descriptions under 155 characters.
  • Pro Tip: "I've consulted for companies spending thousands on SEO tools while their HTML had no semantic structure. Fix the basics first."
